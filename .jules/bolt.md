## 2026-05-18 - SQLite executemany Optimization and Tests Constraints
**Learning:** Python's `sqlite3` driver resets `cur.lastrowid` to `None` after `executemany()` calls. Also, this codebase has a strict regression test (`test_append_batch_timestamps_are_unique_per_row`) requiring every row inserted in a batch to have a unique timestamp, otherwise the test fails.
**Action:** When using `executemany` for batch inserting into `messages`, apply a minor floating-point offset (e.g. `time.time() + (i * 1e-6)`) to generate unique timestamps for each row in the batch. To retrieve the inserted row IDs without a `RETURNING` clause, execute `SELECT last_insert_rowid()` immediately after the `executemany` call, grab the last ID, and calculate the preceding IDs backward based on the batch size `len(params)`.
