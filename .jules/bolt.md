## 2026-07-01 - SQLite executemany Optimization and lastrowid
**Learning:** When optimizing batch inserts with `executemany` in Python's sqlite3, `cur.lastrowid` is set to `None`. This breaks assumptions if the code previously relied on loop-based `execute` where `lastrowid` was populated. Additionally, when batching inserts that require unique timestamps per row, all rows in an `executemany` batch might be constructed too fast, requiring an artificial offset (e.g., `time.time() + (i * 1e-6)`) to preserve row uniqueness constraints.
**Action:** To get IDs from an `executemany` without a `RETURNING` clause, execute `SELECT last_insert_rowid()` immediately after the batch, and calculate the preceding IDs backward using `cur.rowcount` (i.e. `first_id = last_id - cur.rowcount + 1`). Ensure loop-generated parameters apply distinct timestamp offsets when uniqueness is required by schema or tests.
